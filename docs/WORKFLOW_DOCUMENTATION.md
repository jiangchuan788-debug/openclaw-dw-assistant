## 一、智能告警修复系统（每日）

### 1.1 触发条件
- **定时**: 每天早上 06:40 (Asia/Shanghai)
- **手动**: 可随时执行 `repair_strict_7step.py`

### 1.2 执行流程（8步）

```
┌─────────────────────────────────────────────────────────────────────┐
│                        智能告警修复流程                               │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第1步: 扫描告警                                                      │
│ • 从 wattrel_quality_result 表查询未处理告警                          │
│ • 提取表名和 dt 参数                                                 │
│ • 推送钉钉: "发现X个告警，开始处理"                                   │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第2步: 查找工作流位置                                                │
│ • 使用 search_table.py 查找表对应的工作流和任务                       │
│ • 记录映射关系到 JSON 文件                                           │
│ • 生成操作记录目录: auto_repair_records/YYYY-MM-DD/                  │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第3步: 检查工作流状态                                                │
│ • 使用 check_running.py 检查工作流是否空闲                           │
│ • 等待空闲（最多5分钟，每30秒检查一次）                               │
│ • 非空闲则跳过，等待下次执行                                          │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第4步: 执行修复（启动特定任务）                                       │
│ • 限制条件:                                                          │
│   - dt 必须在当前日期 ±10 天内                                       │
│   - 最多并行 2 个任务                                                │
│   - 同一工作流任务串行执行                                           │
│ • 使用 TASK_ONLY 模式启动特定任务                                    │
│   - taskDependType=TASK_ONLY                                         │
│   - startNodeList=<task_code>                                        │
│   - 绝不启动整个工作流！                                             │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第5步: 记录并复验                                                    │
│ • 记录重跑次数到 JSON                                                │
│ • 执行全部 6 个复验工作流（复验项目）                                 │
│   - 每日复验全级别数据(W-1)                                          │
│   - 每小时复验1级表数据(D-1)                                         │
│   - 每小时复验2级表数据(D-1)                                         │
│   - 两小时复验3级表数据(D-1)                                         │
│   - 每周复验全级别数据(M-3)                                          │
│   - 每月11日复验全级别数据(Y-2)                                      │
│ • 再次检查告警状态                                                   │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第6步: 生成钉钉报告                                                  │
│ • 修复成功/失败统计                                                  │
│ • 复验工作流执行结果                                                 │
│ • 告警修复前后对比                                                   │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第7步: 发送TV报告                                                    │
│ • API: https://tv-service-alert.kuainiu.chat/alert/v2/array          │
│ • Bot ID: fbbcabb4-d187-4d9e-8e1e-ba7654a24d1c                      │
│ • 包含简化版修复统计                                                 │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 第8步: 保存操作记录                                                  │
│ • 目录: auto_repair_records/YYYY-MM-DD/                              │
│ • 文件类型:                                                          │
│   - .log  执行日志                                                   │
│   - .json 结构化数据                                                 │
│   - .sh   重跑命令脚本                                               │
│   - .md   操作报告文档                                               │
└─────────────────────────────────────────────────────────────────────┘
```

### 1.3 输入
- **数据库**: `wattrel_quality_result` 表
- **配置**: `config.py` (环境变量)
- **脚本**: `repair_strict_7step.py`

### 1.4 输出
- **钉钉**: 完整修复报告
- **TV**: 简化统计报告
- **本地**: 操作记录文件（4种格式）

### 1.5 关键约束
| 约束 | 值 | 说明 |
|------|-----|------|
| dt 范围 | ±10天 | 超出范围不执行 |
| 最大并行 | 2个任务 | 防止系统过载 |
| 执行模式 | TASK_ONLY | 绝不启动整个工作流 |
| 超时 | 30分钟 | 总执行时间限制 |

## 二、异常调度检测系统（每小时）

### 2.1 触发条件
- **定时**: 每小时第5分钟 (如 00:05, 01:05, 02:05...)
- **手动**: 可随时执行 `auto_stop_abnormal_schedule.py`

### 2.2 执行流程

```
┌─────────────────────────────────────────────────────────────────────┐
│                       异常调度检测流程                                │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 1. 加载调度配置                                                      │
│ • 从 CSV 读取: dolphinscheduler/schedules_export.csv                 │
│ • 包含 28 个定时调度任务配置                                         │
│ • 每天手动更新一次                                                   │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 2. 查询运行中实例                                                    │
│ • API: /projects/{code}/process-instances                            │
│ • 过滤: stateType=RUNNING_EXECUTION                                  │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 3. 遍历检测每个实例                                                  │
│ • 获取实例详情（查看 commandType）                                   │
│ • 只检查 SCHEDULER 类型（调度启动）                                  │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────────────┐
│ 4. 异常判断                                                          │
│ • 异常类型1: 无定时配置但被调度启动                                   │
│   → 工作流Code不在CSV中                                              │
│ • 异常类型2: 调度已下线但仍被启动                                     │
│   → CSV中状态为 OFFLINE                                              │
└─────────────────────────────────────────────────────────────────────┘
                              │
                              ▼
        ┌─────────────────────┴─────────────────────┐
        │                                           │
       正常                                         异常
        │                                           │
        ▼                                           ▼
┌─────────────────┐                   ┌─────────────────────────────┐
│ 不做处理，继续   │                   │ 自动停止实例（无需确认）     │
│ 下一个实例       │                   │ API: executors/execute STOP │
└─────────────────┘                   └─────────────────────────────┘
                                                  │
                              ┌───────────────────┴───────────────────┐
                              │                                       │
                            停止成功                                 停止失败
                              │                                       │
                              ▼                                       ▼
┌─────────────────────────────────────┐   ┌─────────────────────────────────────┐
│ 记录: stopped=True                   │   │ 记录: stopped=False                  │
│ 发送TV: "✅ 已停止: xxx"              │   │ 发送TV: "❌ 停止失败: xxx"            │
│                                      │   │ 标记需人工处理                        │
└─────────────────────────────────────┘   └─────────────────────────────────────┘
```

### 2.3 输入
- **CSV配置**: `dolphinscheduler/schedules_export.csv` (28个调度任务)
- **API**: DolphinScheduler REST API
- **脚本**: `auto_stop_abnormal_schedule.py`

### 2.4 输出

#### TV通知（仅异常时发送）
| 场景 | TV通知 |
|------|--------|
| 成功停止异常实例 | ✅ 发送: "✅ 已停止: xxx" |
| 停止失败 | ✅ 发送: "❌ 停止失败: xxx (需人工处理)" |
| 无异常 | ❌ 不发送 |
| 无运行中实例 | ❌ 不发送 |

#### 钉钉通知（始终发送）
| 场景 | 钉钉通知 |
|------|----------|
| 有异常 | 完整报告（正常/异常统计 + 处理详情） |
| 无异常 | 简洁报告（"✅ 未发现异常调度任务"） |
| 无实例 | 简洁报告（"✅ 无运行中实例"） |

### 2.5 当前28个定时调度

| 状态 | 数量 | 主要工作流 |
|------|------|-----------|
| **ONLINE** | 9个 | H-1, D-1, M-1, T-1, 短信, 业务库校验等 |
| **OFFLINE** | 19个 | DWB, RPT系列, ODS等 |

**CSV更新方式**:
```bash
# 每天晚上手动执行
# 1. 查询最新调度配置
# 2. 更新 dolphinscheduler/schedules_export.csv
```

## 五、关键API端点

| 功能 | 端点 |
|------|------|
| 查询运行中实例 | `/projects/{code}/process-instances?stateType=RUNNING_EXECUTION` |

| 停止实例 | `/projects/{code}/executors/execute` |
