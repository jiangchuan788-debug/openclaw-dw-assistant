## 🔄 执行流程（6步骤）

```
┌─────────────────────────────────────────────────────────────────┐
│ 步骤1: 扫描告警                                                  │
│ - 查询 wattrel_quality_result 表                                │
│ - 筛选 result=1 AND is_repaired=0 AND 3天内                      │
│ - 去重后返回告警列表                                             │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤2: 查找工作流位置                                            │
│ - 优先搜索关键工作流（DWD/DWB/DWS等）                            │
│ - 缓存工作流列表（只获取一次）                                   │
│ - 排除维护任务（补充/删除/清理等关键词）                         │
│ - 匹配任务名或SQL内容                                            │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤3: 启动修复任务                                              │
│ - 遍历找到的告警表                                               │
│ - 调用 DS API 启动工作流实例                                     │
│ - 传递参数: dt, TASK_ONLY模式                                    │
│ - 记录实例ID（处理数组格式）                                     │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤4: 动态监控任务状态                                          │
│ - 每30秒检查一次                                                 │
│ - 查询每个实例的状态                                             │
│ - 失败3次后停止重试                                              │
│ - 所有任务完成后立即退出                                         │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤5: 执行复验工作流                                            │
│ - 记录重跑次数                                                   │
│ - 启动复验工作流（每日全级别 + 3级表）                           │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│ 步骤6: 保存记录并发送TV报告                                      │
│ - 保存JSON详情到文件                                             │
│ - 生成格式化TV报告                                               │
│ - 查询剩余未处理告警数量                                         │
│ - 发送报告到钉钉群                                               │
└─────────────────────────────────────────────────────────────────┘
```

## 🚀 执行方式

### 方式1: 自动定时执行（推荐）

**定时任务配置**:
每天执行 **3次**，分别在：

| 批次 | 时间 | 任务ID | 说明 |
|------|------|--------|------|
| 早上 | 06:40 | 5a02742d-... | 处理夜间积累的告警 |
| 中午 | 12:40 | facfa757-... | 处理上午新增的告警 |
| 晚上 | 18:40 | a0cc07b6-... | 处理下午新增的告警 |

**查看定时任务**:
```bash
openclaw cron list
```

**下次执行时间**:
- 早上批次: 每天 06:40
- 中午批次: 每天 12:40
- 晚上批次: 每天 18:40

**执行间隔**: 每6小时执行一次，确保及时处理告警

## 🔧 故障排查

### 常见问题

#### 1. 定时任务超时

**现象**: `cron: job execution timed out`

**解决**:
- 检查修复任务是否正常完成
- 查看 `/tmp/repair_*.log` 了解执行进度
- 手动执行验证

#### 2. 工作流未找到

**现象**: `❌ 未找到`

**原因**: 
- 表名在DS中没有对应工作流
- 工作流名称不匹配

**解决**:
- 手动在DS中确认工作流存在
- 检查表名是否正确

#### 3. TV报告未发送

**现象**: 钉钉群未收到报告

**解决**:
- 检查网络连接
- 查看日志中的发送状态
- 手动执行测试
